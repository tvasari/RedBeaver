# RedBeaver

A small, dependency-free Python script that generates a blank engagement
folder/file skeleton for a pentest / red team activity. Run it once per
activity, then copy the vulnerability template folder for every finding you
discover.

## Requirements

- Python 3.8+ (no third-party packages required)
- Windows, Linux or macOS

## Quick start

```bash
python redbeaver.py
```

The script will ask three questions. Press Enter on any of them to accept
the default shown in brackets:

```
Client name [Client_Name]:
Output directory [.]:
Name of activity first [Activity_Type_1]:
```

It then creates the structure inside the chosen output directory:

```
Client_Name/
└── 01_Activity_Type_1/
    ├── README.md (not generated by the script itself, add your own)
    ├── Information_Gathering/
    │   ├── Service_Enumeration/
    │   │   └── HTTP/            <- example only, add more protocols as needed
    │   ├── Authentication/
    │   │   ├── found-credentials.txt
    │   │   ├── password-policy.txt
    │   │   └── other.txt
    │   ├── Automated_Scans/
    │   │   ├── nuclei/
    │   │   └── nmap/
    │   ├── Local_Enumeration/
    │   │   └── Host1/
    │   │       └── notes.md
    │   └── Other/
    └── Vulnerabilities/
        └── Vuln_Template/
            ├── notes.md
            ├── <Step>-<Description>.<ext>
            ├── 01-login-page.png
            └── 02-user-a-request.txt
```

After answering the activity question you'll be asked `Add another
activity? [y/N]`, so you can generate several activity folders for the same
client in one run (`02_...`, `03_...`, etc.) without restarting the script.

## Non-interactive usage

Useful for scripting, batch files, or generating several activities at once
without typing anything.

```bash
python redbeaver.py --client "Acme Corp" --activity-name "External Pentest" --output-dir "C:\Engagements"
```

### CLI flags

| Flag              | Description                                                                                                  | Default                |
|-------------------|----------------------------------------------------------------------------------------------------------------|------------------------|
| `--client`        | Client name                                                                                                   | prompts interactively  |
| `--activity-name` | Activity type/name (e.g. `Internal Pentest`). If given, a single activity numbered `01` is created and the "add another activity?" loop is skipped | prompts interactively  |
| `--output-dir`    | Directory in which the `Client_Name` folder is created                                                       | prompts interactively (default shown: `.`) |

Any flag you omit falls back to an interactive prompt (which itself falls
back to the default value on Enter), so you can mix and match, e.g. supply
`--client` and `--output-dir` on the command line and get prompted only for
the activity name.

If the target activity folder already exists and is not empty, the script
warns you and asks for confirmation before writing into it, so you won't
accidentally merge two unrelated runs by mistake.

## Adding a new vulnerability

`Vulnerabilities/Vuln_Template/` is a blank template, not a real finding.
There are no `Discovery` / `Verification` / `Impact` subfolders anymore —
evidence and notes all live directly inside the vulnerability's own folder.
For every vulnerability you discover:

1. Copy `Vuln_Template` into a new folder under `Vulnerabilities/`.
2. Rename it to the vulnerability name (e.g. `Reflected_XSS_Login_Form`).
3. Fill in `notes.md` and drop your evidence files straight into that
   folder.

Evidence files should follow a `<step>-<description>.<ext>` naming
convention so the order of actions is obvious at a glance, e.g.:

```
01-login-page.png
02-user-a-request.txt
03-sqlmap-output.txt
```

The template ships with a `<Step>-<Description>.<ext>` placeholder file plus
two worked examples so the convention is clear at a glance; delete or rename
them once you add your own evidence.

## Local enumeration of compromised hosts

`Information_Gathering/Local_Enumeration/` holds one folder per compromised
local system, each with a `notes.md` pre-filled with sections for system
info, users/privileges, network, installed software/services and general
findings. By default only `Host1/` is created — add more host folders to
the `LOCAL_ENUMERATION_HOSTS` config list (see below) if you know upfront
you'll be enumerating several boxes, or just copy `Host1/` by hand for
subsequent hosts during the engagement.

## Customizing the template

Everything you're likely to want to change lives in one config block near
the top of `redbeaver.py` — no need to touch the rest of the script.

### Add a tool to Automated_Scans

```python
AUTOMATED_SCAN_TOOLS = [
    "nuclei",
    "nmap",
    "ffuf",          # <- just add a new entry
]
```

### Add a protocol under Service_Enumeration

Only one example protocol folder (`HTTP`) is created out of the box, to
keep the skeleton lean. Add the ones relevant to your engagement:

```python
SERVICE_ENUMERATION_PROTOCOLS = [
    "HTTP",
    "SSH",
    "SMB",
    "LDAP",          # <- just add a new entry
]
```

### Add a host under Local_Enumeration

```python
LOCAL_ENUMERATION_HOSTS = [
    "Host1",
    "Host2",         # <- just add a new entry
]
```

### Add a brand new top-level section

Sections are defined in `build_structure()` as a nested dict. Each level
supports:

- `"_dirs": [names...]` — plain empty subfolders
- `"_files": {filename: content}` — files, optionally pre-filled with text
- any other key — a nested subfolder, recursed into with the same rules

Example: adding a `Reporting/` section as a sibling of
`Information_Gathering` and `Vulnerabilities`:

```python
def build_structure() -> dict:
    return {
        "Information_Gathering": { ... },
        "Vulnerabilities": { ... },
        "Reporting": {
            "_dirs": ["Drafts", "Final"],
        },
    }
```

### Change default values

```python
DEFAULT_CLIENT_NAME = "Client_Name"
DEFAULT_ACTIVITY_NAME = "Activity_Type"
DEFAULT_OUTPUT_DIR = "."
```

### Change the vuln template's example files or notes.md content

```python
VULN_EXAMPLE_FILES = [
    "<Step>-<Description>.<ext>",
    "01-login-page.png",
    "02-user-a-request.txt",
]

VULN_NOTES_TEMPLATE = """# {vuln_name}

## Summary
...
"""
```

### Change the Local_Enumeration notes.md content

```python
LOCAL_ENUMERATION_NOTES_TEMPLATE = """# {host_name} - Local Enumeration

## System Info
...
"""
```

## Notes on folder/file naming

Windows doesn't allow the characters `< > : " / \ | ? *` in file or folder
names. Whatever you type for the client name or activity name is
automatically stripped of these characters before folders are created, so
you don't need to sanitize input yourself. This also applies to the
built-in example filenames (e.g. `<Step>-<Description>.<ext>` is written to
disk as `Step-Description.ext`).

## Possible extensions (not implemented)

- A flag to auto-create a named vulnerability folder from `Vuln_Template`
  (e.g. `--new-vuln "Reflected XSS"`) instead of copying it by hand.
- A flag to auto-create a new `Local_Enumeration` host folder
  (e.g. `--new-host "DC01"`) instead of copying `Host1/` by hand.
- An `--activity-number` flag to explicitly set the starting activity
  number instead of always starting at `01`.
- Loading the config block from an external JSON/YAML file instead of
  editing the Python script directly, for sharing presets across a team.
- Auto-initializing a git repo in the activity folder for change tracking.
